# Project Steward Optimization Plan

This document describes the optimization plan for Project Steward after the 1.1.4 release. It focuses on performance, maintainability, AI session extensibility, and future product velocity.

## Background

Project Steward has evolved from a project dashboard into a sidebar-first project hub with local, SSH, Dev Container, Codespaces, and AI session workflows. Recent releases added Codex, Kimi, and Claude session discovery, session terminal resume, session pinning, aliases, context menus, sticky sidebar groups, and GitHub release automation.

The feature set is now useful, but the implementation has accumulated several pressure points:

- `src/dashboard.ts` has grown to more than 3,500 lines and owns too many responsibilities.
- AI session support is implemented with repeated provider-specific branches.
- Session updates still refresh the entire webview HTML.
- Several hot paths use synchronous filesystem or shell operations on the extension host.
- Some UI behavior lives in generated HTML strings and imperative DOM scripts, making small changes risky.
- Lint warnings are accepted as warnings, which reduces confidence when making larger changes.

The goal of this optimization work is to make the extension smoother, safer to extend, and easier to maintain without changing the core user experience.

## Goals

1. Reduce sidebar CPU and rendering work during AI session activity.
2. Make adding a new AI provider much cheaper and less error-prone.
3. Break large modules into testable, focused units.
4. Avoid blocking the VS Code extension host with avoidable synchronous work.
5. Preserve existing user data and behavior.
6. Keep release risk low by splitting implementation into reviewable PRs.

## Non-Goals

- Rewriting the UI in React, Vue, Svelte, or another frontend framework.
- Changing the saved project data format unless necessary.
- Changing marketplace identity, command IDs, or user-facing branding.
- Replacing the existing sidebar with an editor webview again.
- Implementing a paid feature system or monetization.

## Current Architecture Snapshot

### Extension Host

The extension host entry point is `src/dashboard.ts`. It currently handles:

- Extension activation and command registration.
- Sidebar webview provider lifecycle.
- Project CRUD and group CRUD.
- Project open/save behavior.
- Remote URI resolution for SSH, Dev Containers, Codespaces, and WSL-like cases.
- Favorites and Open Project virtual groups.
- AI session discovery assignment.
- AI session terminal creation and reuse.
- AI session pin, alias, archive, rename, and context menu actions.
- Webview message dispatch.
- Git repository detection.
- Error logging.

This makes the module hard to reason about. A change in one area often requires scanning unrelated logic.

### Webview

The webview is generated by `src/webview/webviewContent.ts` and behavior is split across:

- `src/webview/webviewProjectScripts.js`
- `src/webview/webviewDnDScripts.js`
- `src/webview/webviewFilterScripts.js`
- `media/styles.scss`

The extension currently updates the sidebar by assigning a new string to `webview.html`. This recreates the document, reloads scripts, rebuilds DOM nodes, and reinitializes event handlers.

### AI Session Services

AI session providers are currently represented by separate services:

- `src/services/codexSessionService.ts`
- `src/services/kimiSessionService.ts`
- `src/services/claudeSessionService.ts`

They expose similar concepts:

- `getSessions(forceRefresh?)`
- `watchSessionChanges(onDidChange)`
- `archiveSession(sessionId)`
- cache invalidation
- filesystem discovery
- session metadata parsing

But the common contract is implicit. Provider behavior is wired together manually in `dashboard.ts`.

## Problems And Opportunities

## 1. Full Webview Rebuilds For Session Updates

### Current Behavior

`SidebarStewardViewProvider.refresh()` rebuilds the entire webview:

```ts
this._view.webview.html = getStewardContent(...)
```

This is simple, but expensive. It reloads scripts and recreates DOM even when only the session list changed.

### Impact

- Session file activity can cause visible UI churn.
- Search state, scroll state, expanded state, and hover state are easier to disturb.
- The webview repeatedly reinitializes drag/drop, filtering, context menu, and sticky header logic.
- Performance cost scales with saved project count and session count.

### Proposed Direction

Move AI session updates to an incremental webview message:

```ts
webview.postMessage({
  type: 'ai-sessions-updated',
  version: 1,
  sequence: 12,
  openProjects: [...]
});
```

The webview should update only the Open Project session sections.

### Message Contract

Incremental updates must use an explicit, versioned message shape:

```ts
interface AiSessionsUpdatedMessage {
  type: 'ai-sessions-updated';
  version: 1;
  sequence: number;
  generatedAt: string;
  openProjects: OpenProjectAiSessionViewModel[];
}

interface OpenProjectAiSessionViewModel {
  projectId: string;
  projectKey: string;
  activeProvider: AiSessionProviderId;
  expanded: boolean;
  providers: AiSessionProviderSummary[];
  sessionsByProvider: Record<AiSessionProviderId, AiSessionViewModel[]>;
  unavailableProviders: AiSessionProviderId[];
}
```

Rules:

- The extension host increments `sequence` every time it sends AI session data.
- The webview ignores messages with a sequence older than the latest applied message.
- The webview treats missing project IDs as deleted Open Project cards and removes their session sections.
- The webview preserves local DOM state that is intentionally local, such as current scroll position inside the session list where possible.
- The extension host remains the source of truth for expanded state, active provider, pinned sessions, and aliases until state ownership is explicitly changed.
- Search indexing must be rebuilt for updated Open Project cards after applying the message.
- A full refresh remains the fallback when the webview reports an unsupported message version or a missing DOM target.

### Suggested Implementation

1. Add a `postMessage` helper to the sidebar provider.
2. Add a serializable model for Open Project AI session data.
3. Render Open Project session rows through a reusable webview-side function.
4. On session watcher changes:
   - Recompute open project session data.
   - Send `ai-sessions-updated`.
   - Do not rebuild full HTML.
5. Keep full refresh for:
   - project/group mutations
   - configuration changes
   - extension visibility restoration
   - error recovery

### Acceptance Criteria

- Active AI session writes no longer replace the whole webview document.
- Search bar stays visible and stable.
- Expanded Open Project cards remain expanded.
- Provider selection, pinned state, aliases, and archive behavior still work.
- New sessions appear without manually reloading the view.
- Out-of-order session update messages are ignored.
- Search results include renamed or newly discovered sessions after an incremental update.

## 2. Provider Registry For AI Sessions

### Current Behavior

Provider-specific logic is repeated in many places:

- Codex/Kimi/Claude services are instantiated separately.
- Terminal maps are separate.
- In-flight resume sets are separate.
- Resume command builders are separate.
- New session command builders are separate.
- Archive dispatch uses multiple switch cases.
- Provider display data is spread across webview and dashboard logic.

### Impact

Adding a fourth provider requires touching many unrelated places. This increases regression risk.

### Proposed Direction

Introduce an `AiSessionProvider` registry.

Example interface:

```ts
export interface AiSessionProvider {
  id: AiSessionProviderId;
  label: string;
  terminalNamePrefix: string;
  terminalEnvKey: string;
  markerDirName: string;
  service: AiSessionService;
  getTerminalCwd(session: CodexSession, project: Project): string;
  buildResumeCommand(args: AiSessionCommandArgs): string;
  buildNewSessionCommand(args: AiSessionCommandArgs): string;
}
```

Example service contract:

```ts
export interface AiSessionService {
  getSessions(options?: AiSessionQueryOptions): AiSessionReadResult;
  watchSessionChanges(onDidChange: () => void): Disposable;
  archiveSession(sessionId: string): boolean;
  invalidateCache(): void;
}

export interface AiSessionQueryOptions {
  forceRefresh?: boolean;
  candidatePaths?: string[];
}
```

The initial implementation can ignore `candidatePaths` and keep current behavior. The options object is still preferable because it lets later performance work narrow provider scans without changing the interface again.

### Suggested Implementation

1. Add `src/aiSessions/types.ts`.
2. Add `src/aiSessions/providers.ts`.
3. Move command builders into provider definitions.
4. Replace provider-specific maps with generic maps:

```ts
const aiSessionTerminals = new Map<AiSessionProviderId, Map<string, AiSessionTerminalEntry>>();
const aiSessionResumesInFlight = new Map<AiSessionProviderId, Set<string>>();
```

5. Replace switch chains with provider lookup:

```ts
const provider = getAiSessionProvider(providerId);
```

6. Keep `AiSessionProviderId = 'codex' | 'kimi' | 'claude'` for now.

### Acceptance Criteria

- Codex, Kimi, and Claude behavior remains unchanged.
- Adding a provider mostly means adding one provider definition plus one session service.
- Terminal reuse, marker files, archive, create, resume, pin, and alias behavior still pass manual testing.

## 3. Split `dashboard.ts`

### Current Behavior

`dashboard.ts` is the central module for almost everything.

### Impact

- Hard to review.
- Hard to test.
- Hard to isolate regressions.
- Easy to create accidental coupling.

### Proposed Module Split

#### `src/stewardViewProvider.ts`

Owns:

- `SidebarStewardViewProvider`
- webview lifecycle
- full refresh
- partial postMessage helpers
- visibility tracking

#### `src/aiSessions/aiSessionController.ts`

Owns:

- session provider registry orchestration
- session assignment to Open Projects
- pinned/alias pruning
- watcher lifecycle
- incremental session refresh messages

#### `src/aiSessions/aiSessionTerminalService.ts`

Owns:

- terminal creation
- terminal reuse
- terminal marker paths
- terminal completion detection
- resume/new command dispatch

#### `src/projects/openProjectService.ts`

Owns:

- Open Project virtual group creation
- saved project matching
- current workspace folder/workspace file detection
- open project card metadata

#### `src/projects/remoteProjectResolver.ts`

Owns:

- Dev Container URI resolution
- SSH URI resolution
- Codespaces URI resolution
- recent entries fallback

#### `src/storage/aiSessionAliasStore.ts`

Owns:

- alias file path
- alias read/write

- alias pruning support

### Suggested Split Order

1. Extract pure helper functions first.
2. Extract shared AI session types and view models.
3. Extract provider registry.
4. Extract AI session terminal logic.
5. Extract Open Project logic.

### Acceptance Criteria

- No behavior changes in each extraction PR unless explicitly intended.
- Test compile and lint pass after each step.

## 4. Synchronous I/O And Extension Host Blocking

### Current Hotspots

#### Git Detection

`isFolderGitRepo()` uses `execSync`:

```ts
execSync(`cd ${fPath} && git rev-parse --is-inside-work-tree`)
```

This can block the extension host and has shell quoting risk.

#### Session Discovery

Session services use synchronous filesystem operations. Some of this is acceptable because the code is simpler and now only runs while the sidebar is visible, but large session directories can still cause noticeable work.

#### Alias Storage

Alias read/write uses synchronous filesystem calls. This is low volume but should eventually move behind a small store abstraction.

### Proposed Direction

1. Replace `execSync` git detection with one of:
   - cached async `git rev-parse` with safe `spawn`
   - VS Code Git extension API, if available and reliable enough
   - async `vscode.workspace.fs.stat`
   - `existsSync(path.join(folder, '.git'))` only as a fast approximate check
2. Cache `isGitRepo` for saved projects and open projects.
3. Move alias storage to a store class so future async conversion is contained.
4. Add provider-level session scan metrics in debug logs.

Important: checking for a `.git` directory is not equivalent to `git rev-parse`. It can miss worktrees, submodules, nested repositories, and folders where `.git` is a file. If the UI needs correctness, prefer cached async `git rev-parse`. If the UI only needs a rough local-folder icon hint, the `.git` check can be used as a cheap fallback.

### Acceptance Criteria

- Opening the sidebar does not synchronously invoke shell commands for every project.
- Dev Container and remote URI projects do not trigger local git shell checks.
- Project cards still show expected icons and git metadata where supported.

## 5. AI Session Discovery Performance

### Codex

Codex currently reads session index entries and first-line metadata from JSONL files. This is relatively efficient because it avoids reading full session files.

Potential improvements:

- Cache `sessionFiles` signature and parsed metadata separately.
- Avoid repeated recursive directory traversal during archive operations.
- Consider configurable `CODEX_HOME` changes while visible.

### Kimi

Kimi scans configured work dirs and checks state/wire files. This is moderate cost when many work dirs exist.

Potential improvements:

- Cache work dir hash mapping.
- Track only work dirs relevant to currently open projects when the sidebar is open.
- Avoid building signatures for unrelated work dirs if possible.

### Claude

Claude currently reads full JSONL files on cache miss to find metadata and title candidates.

Potential improvements:

- Read the first N lines for `cwd`.
- Read the last N KB for recent title/prompt candidates.
- Keep per-file cache keyed by size and mtime.
- Stop scanning old project directories unrelated to current Open Projects when possible.

### Provider-Aware Scope

The session services currently return all available sessions. For sidebar display, only sessions matching current open projects are useful.

Future interface:

```ts
getSessions(options?: {
  candidatePaths?: string[];
  forceRefresh?: boolean;
}): AiSessionReadResult;
```

This allows providers to narrow work when possible.

### Acceptance Criteria

- Large Claude session files do not require full reads during normal refresh.
- Kimi scanning cost scales with open projects where possible.
- Newly created sessions still appear within a reasonable time.

## 6. Webview UI Maintainability

### Current Behavior

HTML generation and behavior are split across TypeScript template strings and imperative JavaScript. This is workable but can become fragile as the UI grows.

### Proposed Direction

Keep the existing no-framework approach, but introduce small rendering helpers:

- `renderOpenProjectsGroup(data)`
- `renderAiSessionSection(project)`
- `renderAiSessionRow(session)`
- `updateAiSessionSection(projectId, data)`

Move these helpers into `src/webview/webviewProjectScripts.js` or a new webview script file.

### Benefits

- Enables incremental updates.
- Keeps server-side and client-side markup consistent.
- Reduces full HTML string churn.
- Makes it easier to test UI snippets manually.

### Acceptance Criteria

- Initial render and incremental render use the same row structure.
- Context menu, pin, archive, resume, provider selection, and create actions continue to work.
- Search indexing includes updated session names after incremental updates.

## 7. State Ownership Cleanup

### Current State Types

State is spread across:

- `globalState` for project data, pinned sessions, expanded projects, provider selection.
- extension storage file for AI session aliases.
- DOM/sessionStorage for search.
- provider-specific session files outside the extension.

### Issues

- Some state should be window-local but is global.
- Some state should be machine-local and not synced.
- Some state should be durable but local to the extension host.

### Proposed Ownership Rules

| State | Owner | Sync? | Notes |
| --- | --- | --- | --- |
| Saved projects/groups | `globalState` or settings | Yes, existing behavior | Preserve compatibility |
| Favorites | project data | Yes | User expects it to follow project list |
| Collapse all | webview DOM | No | Already local |
| Individual group collapsed | existing group state/globalState | Existing behavior | Can revisit later |
| Open Project expanded sessions | extension storage file or webview state | No | Should become local-only; current `globalState` usage is transitional |
| Active AI provider per project | extension storage file | No | Should be local-only and not synced with project data |
| Pinned AI sessions | extension storage file | No | User requested minimal cross-machine sync pressure |
| AI aliases | extension storage file | No | Already local |
| Search text | sessionStorage | No | Good |

### Acceptance Criteria

- No AI session local UI state is written into saved project data.
- AI session aliases, pinned sessions, and provider preferences are local to the extension host.
- Any remaining `globalState` usage for AI session UI state is documented as transitional and has a migration path.
- State that should be local is documented and easy to migrate later.

## Risks

### Incremental Webview Updates Can Drift From Initial Render

If initial HTML and incremental render paths diverge, session rows may look or behave differently after updates.

Mitigation:

- Share rendering helpers where possible.
- Keep full refresh fallback.
- Test pin/archive/rename/provider selection after an incremental update.

### Provider Registry Can Become Too Abstract

Over-abstraction can make simple provider behavior harder to follow.

Mitigation:

- Keep provider interface small.
- Move only repeated behavior into shared code.
- Allow provider-specific parsing and command building to stay provider-owned.

### Async Refactors Can Change Ordering

Moving synchronous work async can introduce races in refresh and terminal actions.

Mitigation:

- Make refresh requests idempotent.
- Keep in-flight sets for terminal operations.
- Add timestamps or sequence IDs for webview session update messages if needed.

### Remote And Dev Container URI Handling Is Fragile

Remote URI resolution depends on VS Code internals and recent entries.

Mitigation:

- Extract without changing behavior first.
- Add fixtures or logged examples for known local, SSH, Dev Container, and Codespaces cases.

## Recommended Next Step

Start with Phase 0: safety net and message protocol.

This is the safest foundation because the next changes touch terminal behavior, provider routing, and webview update semantics. The first code PR should extract provider-neutral types and pure helpers without behavior changes. After that, the provider registry can be introduced on top of a smaller, testable surface, and incremental webview updates can depend on a clear message contract instead of ad hoc DOM patching.
